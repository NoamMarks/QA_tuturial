1. מה זה Postman ואיך מבינים API?

💡 הסבר פשוט: אנלוגיית המלצר

הרבה מתחילים מסתבכים עם המושג API. תחשבו על זה כמו במסעדה:

  • הלקוח (Client): זה אתם (או האפליקציה). אתם יושבים בשולחן ורוצים מידע (אוכל).
  • המטבח (Server/Database): שם נמצאים כל הנתונים וחומרי הגלם. אתם לא יכולים להיכנס למטבח ולקחת לבד.
  • המלצר (API): הוא המתווך! אתם נותנים לו בקשה (Request), הוא הולך למטבח, וחוזר אליכם עם התשובה (Response).

Postman הוא הכלי שמאפשר לנו "לדבר עם המלצר" ישירות, לבדוק אם הוא מביא את המנה הנכונה, ואם הוא מחזיר תשובה מהר.

מבחינה טכנית, Postman הוא כלי גרפי (GUI) לניהול, בדיקה והרצת קריאות HTTP. הוא מאפשר לנו לוודא שהצד האחורי (Backend) של המערכת עובד תקין עוד לפני שיש לנו ממשק משתמש מוכן.

2. התקנה ויצירת חשבון

  1. היכנסו ל־postman.com והורידו את הגרסה המתאימה למערכת ההפעלה.
  2. התקינו את התוכנה. מומלץ ליצור חשבון חינמי כדי שהבדיקות שלכם ישמרו בענן.

3. היכרות עם הממשק

  • סרגל צד (Sidebar): "התיקיות שלי". כאן תמצאו את האוספים (Collections) והיסטוריית הקריאות.
  • החלונית הראשית (Main Panel): אזור העבודה. כאן בונים את הבקשה (כתובת ה-URL, סוג הבקשה) ורואים את התשובה.
  • קונסול (Console): נמצא למטה בצד שמאל. מציג לוגים טכניים של מה שקרה "מאחורי הקלעים".

4. יצירת קריאת API ראשונה

  1. בחרו את HTTP Method (למשל GET כדי לקבל מידע, POST כדי ליצור משהו חדש).
  2. הזינו את כתובת ה־URL של ה־API (למשל: https://reqres.in/api/users).
  3. לחצו על הכפתור Send.
  4. למטה בחלק של ה-Response, תראו את התשובה שחזרה מהשרת (לרוב בפורמט JSON) ואת קוד הסטטוס (למשל 200 OK).

5. ארגון קריאות באוספים (Collections)

כדי לא לאבד את הבדיקות שלכם, שמרו אותן בתוך Collection. זה כמו תיקייה ראשית לפרויקט. מומלץ ליצור תיקיות משנה לפי נושאים (למשל: "ניהול משתמשים", "הזמנות", "התחברות").

6. סביבות עבודה (Environments)

בעולם האמיתי, יש לנו כמה סביבות: פיתוח (Dev), בדיקות (QA) וייצור (Prod). הכתובות שונות בכל סביבה.

במקום לשנות את הכתובת ידנית כל פעם:

  1. צרו Environment חדש (למשל "QA Env").
  2. הגדירו משתנה בשם base_url עם הכתובת של סביבת הבדיקות.
  3. בבקשה עצמה, כתבו {{base_url}}/users. כעת תוכלו להחליף סביבה בקליק אחד למעלה בצד ימין.

7. כתיבת בדיקות אוטומטיות (Tests)

זה הכוח האמיתי של Postman. בלשונית Tests אפשר לכתוב קוד קטן שיבדוק את התשובה בשבילכם. 

// בדיקה שהשרת החזיר תשובה תקינה (200)
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

// בדיקה שהתגובה מהירה (פחות מ-200 מילי-שניות)
pm.test("Response time is less than 200ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(200);
});

8. סקריפטים לפני שליחה (Pre-request Script)

כאן כותבים קוד שרץ לפני שהבקשה נשלחת. שימושי מאוד ליצירת תאריכים עדכניים או מספרים רנדומליים. 

// יצירת תאריך של היום ושמירה למשתנה
pm.environment.set("timestamp", new Date().toISOString());

9. הרצת אוספים (Collection Runner)

רוצים להריץ 50 בדיקות ברצף? אל תעשו את זה ידנית. השתמשו ב-Runner. הוא יריץ את כל התיקייה, יבדוק את כל ה-Tests שכתבתם, ויפיק דוח ירוק/אדום בסוף.

10. עבודה עם Newman

Newman הוא הכלי שמאפשר להריץ את הטסטים של Postman דרך שורת הפקודה (CMD). זה קריטי לשילוב בתוך מערכות אוטומציה (CI/CD) כמו Jenkins או Azure DevOps. 

newman run MyCollection.json -e MyEnv.json

מילון מונחים

API
ממשק שמאפשר לתוכנות לדבר אחת עם השנייה.
JSON
השפה הנפוצה ביותר לתקשורת ב-API. נראה כמו רשימה של מפתח:ערך.
Status Code
המספר שהשרת מחזיר. 200=הצלחה, 400=טעות שלכם, 500=טעות של השרת.
Collection
אוסף של קריאות API, שקול ל"פרויקט בדיקות".
