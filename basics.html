1. חשיבות בדיקות תוכנה

בדיקות תוכנה אינן “מרצון” אלא שלב הכרחי בכל פרויקט פיתוח:

  • חיסכון בעלויות תיקון – זיהוי באגים מוקדם חוסך 10–100 פעמים את עלות התיקון בשלב הייצור.
  • הבטחת איכות ויציבות – מערכת שנבדקה היטב נחשבת לאמינה, ומשפרת את שביעות רצון המשתמש.
  • מניעת סיכונים עסקיים – מוצר תקול עלול לגרום להפסדים כספיים, תביעות ופגיעה במוניטין.
  • שיפור מתמיד בתהליך הפיתוח – משוב עסקי וטכני דרך בדיקות מובנות יוצר תהליך איכותי.

2. מטרות הבדיקות

  1. איתור באגים (Defect Detection) – זיהוי שגיאות לוגיות ותקלות לפני שחרור.
  2. מניעת באגים (Defect Prevention) – ביקורות סטטיות מונעות הכנסת באגים.
  3. אימות דרישות (Requirement Verification) – הבטחה שהמוצר עומד בדרישות.
  4. הערכת איכות (Quality Assessment) – מדידת כיסוי בדיקות, כמות תקלות ויציבות.
  5. תמיכה בהחלטות שחרור – קבלת החלטה מבוססת על נתוני בדיקות.

4. שבעת עקרונות הבדיקות

  1. בדיקות מגלות נוכחות באגים – אי־אפשר להוכיח שאין באגים.
  2. בדיקה מלאה בלתי אפשרית – צריך דגימות חכמות.
  3. בדיקה מוקדמת – עלות תיקון נמוכה יותר.
  4. צבירת באגים (Defect Clustering) – 20% מהקוד מכיל 80% מהבאגים (Pareto).
  5. Pesticide Paradox – יש לעדכן בדיקות כדי לגלות באגים חדשים.
  6. בדיקה תלויה בהקשר – סוג הפרויקט קובע את סוג הבדיקות.
  7. Absence‑of‑Errors Fallacy – עמידה בדרישות לא מבטיחה מוצר שימושי.

5. רמות בדיקה

5.1 בדיקת יחידה (Unit Testing)

בדיקה של פונקציות או מחלקות בודדות, מבוצעת על־ידי מפתחי התוכנה עם כלים כמו JUnit, pytest.

5.2 בדיקת אינטגרציה (Integration Testing)

בדיקה של תקשורת בין מודולים ושירותי API; טיפוסי שיטות: Big Bang, Top‑Down, Bottom‑Up.

5.3 בדיקת מערכת (System Testing)

בדיקה מקצה‑לקצה של המערכת כולה בסביבת סימולציה של ייצור.

5.4 בדיקת קבלה (Acceptance Testing)

בדיקה תחת פיקוח הלקוח או משתמשי קצה; סוגים: Alpha, Beta, Contract, Regulatory.

6. מחזור חיי הבדיקה

  1. תכנון & בקרת בדיקות – קביעת מטרות, טווח, לוחות זמנים ומשאבים.
  2. ניתוח & תכנון – זיהוי תרחישי בדיקה, כתיבת Test Cases.
  3. הטמעה & הרצה – הכנת סביבה, הרצת Test Cases ותיעוד תוצאות.
  4. הערכת קריטריוני יציאה & דיווח – בדיקת עמידה במטרות והכנת דו"ח מסכם.
  5. פעילויות סגירה – הפקת לקחים, ארכיון Testware וסגירת Test Plan.

8. שיטות וגישות בבדיקה

8.1 בדיקה “קופסה שחורה” (Black‑Box Testing)

מתמקדת בפונקציונליות בלבד, ללא גישה לקוד המקור.

  • Equivalence Partitioning (חלוקת משבצות)
    מחלקים קבוצת קלטים למשבצות (classes) שבהן התגובה זהה. דוגמה: שדה גיל (0–120) מחולק ל־
    • גיל תקין (0–120) → בוחרים 0, 60, 120
    • גיל מתחת לטווח (<0) → -1, -5
    • גיל מעל לטווח (>120) → 121, 200
  • Boundary Value Analysis (ניתוח ערכי קצה)
    בודקים את ערכי הקצה של כל משבצת, מאחר ובאגים מתרכזים שם. תרחישי בדיקה לדוגמה: 0, 1, 119, 120, -1, 121
  • Decision Table Testing (בדיקות טבלת החלטה)
    בונים טבלה שממפה תנאים לפעולות כדי לכסות כל הקומבינציות. טבלה לדוגמה:
    מחובר?מוסמך?פעולה
    כןכןגישה מלאה
    כןלאגישה מוגבלת
    לאכןפנה להתחברות
    לאלאסרב גישה
  • State Transition Testing (בדיקות מעבר מצב)
    מתבססים על דיאגרמת מצבים ומעברים. תרחיש לדוגמה: טלפון חכם – מצבים: כבוי, נעול, פתוח; מעברים: on, lock, unlock.
  • Use Case Testing (בדיקות תרחישי שימוש)
    בונים תרחישי בדיקה על בסיס “סיפור משתמש” (User Story). תרחיש לדוגמה: “הוסף מוצר לעגלה → עדכן כמות → בצע תשלום” כולל תרחישי קצה (כרטיס פג תוקף).

8.2 בדיקה “קופסה לבנה” (White‑Box Testing)

מתמקדת במבנה הקוד ובזרימת הלוגיקה:

  • Statement Coverage (כיסוי שורות) – כל שורת קוד רצה לפחות פעם אחת.
  • Branch Coverage (כיסוי ענפים) – כל תנאי (if/else) נבדק בשני הענפים.
  • Path Coverage (כיסוי מסלולים) – כל המסלולים האפשריים בגרף השליטה.
  • Condition Coverage (כיסוי תנאים) – כל ביטוי לוגי נבדק בכל הקומבינציות.

8.3 בדיקות מבוססות ניסיון (Experience‑Based Testing)

  • Exploratory Testing (בדיקות סקרניות) – הבודק חוקר את המערכת ללא תסריט קבוע, מגלה באגים בלתי צפויים.
  • Error Guessing (ניחוש שגיאות) – הבודק מנחש היכן עשויים להופיע באגים לפי ניסיון.
  • Checklist‑Based Testing (בדיקות מבוססות רשימת בדיקה) – שימוש ברשימת פריטים (checklist) כדי לוודא כיסוי נושאים מרכזיים.

8.4 בדיקות מבוססות סיכון (Risk‑Based Testing)

ממפים רכיבים לפי רמת סיכון (השפעה והסתברות לכישלון) – ומקדישים משאבים לרכיבים הקריטיים ביותר.

8.5 בדיקות סטטיות (Static Testing)

  • Reviews & Walkthroughs (סקירות וסיורי בדיקה) – בדיקה ידנית של דרישות, עיצוב וקוד ללא הרצה.
  • Inspections (בדיקות פורמליות) – תהליך פורמלי עם תפקידים (Moderator, Reader, Recorder) ודיווח מסודר.
  • Static Analysis Tools (כלים לניתוח סטטי) – כלים אוטומטיים (כמו SonarQube, ESLint) לזיהוי בעיות לפני הרצה.

9. תיעוד בדיקות

  1. Test Plan – אסטרטגיה, טווח, סיכונים.
  2. Test Strategy – קווים מנחים ארגוניים.
  3. Test Case Specification – פירוט תרחישים.
  4. Traceability Matrix – כיסוי דרישות.
  5. Test Incident Report – תיעוד מפורט של באגים.
  6. Test Summary Report – סיכום תוצאות ומדדים.

10. מדדי בדיקה ו‑KPIs

  • Requirement Coverage: אחוז דרישות מכוסות.
  • Defect Density: באגים ל‑KLOC.
  • Defect Leakage: באגים שברחו לסביבת הייצור.
  • MTTD / MTTR: זמני זיהוי ותיקון ממוצעים.
  • Test Execution Rate: מהירות הרצת Test Cases.
